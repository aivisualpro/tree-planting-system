# Internationalization (i18n) Strategy

This document explains the two-tier dynamic multi-language system used across Flutter and Nuxt.

## Tier 1 — App-shell strings (compile-time)

Used for static UI text like "Login", "Save", "Cancel", and navigation labels.

- **Source of Truth**: `shared/i18n/*.json` (e.g., `en.json`, `es.json`)
- **Nuxt**: Uses `@nuxtjs/i18n` to load JSON files directly from `shared/i18n/locales/*.json`.
- **Flutter**: ARB files are generated into `mobile/lib/l10n/` via a sync script (`shared/scripts/sync-i18n.js`). The Flutter app uses `flutter gen-l10n` to build Dart localization classes.

**Developer Workflow (Adding an App-shell string)**:
1. Add the key-value pair to `shared/i18n/en.json` (and others if available).
2. Run `node shared/scripts/sync-i18n.js`.
3. Rebuild the Flutter app to generate the new Dart accessors.
4. Note: Changes require a code release to reach end-users.

## Tier 2 — Entity translations (runtime, from DB)

Used for dynamic data such as country names, tree species names, and other master data entities.

- **Storage**: `translations` table in the database.
  Schema: `id, entity_type, entity_id, field, locale, value, updated_at`
- **Management**: Editable at runtime by administrators via the Nuxt admin UI (`/masters/translations`). No code deploy is needed.
- **Flutter**: Cached locally in Drift (`Translations` table). Falls back to network fetch on miss. Background sync pulls new translations on startup, and Supabase Realtime subscriptions invalidate cache upon change.
- **Nuxt**: Handled via `useEntityTranslation` composable and cached in a Pinia store.

**Admin Workflow (Adding a new language)**:
1. Navigate to `/masters/translations` in the admin UI.
2. Click "Add Language".
3. Provide the locale code (e.g., `pt`) and display name (e.g., `Portuguese`).
4. The system automatically creates `null` rows for all entities for the new locale based on the `en` base.
5. Admins can progressively fill in the translations.
6. The new language immediately becomes available in the app via API requests and background sync.

## Fallback Chain

When requesting a translation for a specific `locale`:
1. Check for exact match in `target_locale`.
2. Fallback to `en` (English) translation.
3. Fallback to the raw `field` or entity identifier as a last resort.

## Caching Strategy

- **Flutter**: Offline-first via Drift.
- **Nuxt**: In-memory caching via Pinia to avoid repeated API calls during a single session.
