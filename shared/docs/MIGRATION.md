# AppSheet to Supabase Migration Plan

## Coexistence & Cutover Plan

### Week -2: Preparation
- Freeze all schema changes on the AppSheet application.
- The client team executes full CSV exports of all operational tables.
- Provision Supabase production and staging environments.

### Week -1: Dry Run
- Perform an end-to-end dry run of the migration pipeline (`pnpm migrate:all`) against the **staging** Supabase environment.
- Validate CSV ingestion, deduplication rules, and translation generation.
- Client teams review `reports/verification.md` and `reports/rejects.json`.
- Fix any identified parsing, formatting, or Zod validation issues in the data or transformation scripts.

### Week 0: Pilot Release
- Execute a final CSV export for **1 target pilot country** (e.g., Tanzania).
- Run the migration specifically scoped or verified for the pilot country into **production**.
- Launch the new Flutter mobile app and Nuxt web admin for the pilot country only.
- AppSheet remains the primary system for all other countries.

### Week 1: Stabilization
- Monitor the pilot country's data flow.
- Resolve any bugs or sync issues identified during the pilot.
- Conduct training sessions for 5 key coordinators representing the upcoming country rollouts.

### Week 2-3: Rolling Migration Waves
- Migrate remaining countries in structured waves (e.g., Wave 1: Malawi, Wave 2: Thailand).
- Before each wave, a fresh CSV export is processed through the migration pipeline (`onConflict` upsert handles existing records idempotently).
- Users in those countries are progressively onboarded via the `07-invite-users.ts` script.

### Week 4: Full Cutover
- AppSheet application is set to read-only mode.
- Final delta sync migration is run for any straggler records.
- 100% of global operations run on the new Supabase + Flutter/Nuxt stack.

## Rollback Strategy
If critical failures occur during or immediately after cutover:
1. **AppSheet Reversion:** The AppSheet application remains functional and holds the last exported state (retained for 90 days). It can be toggled back to write-mode instantly.
2. **Supabase PITR:** The Supabase Pro plan provides 7-day Point-In-Time Recovery. We can instantly restore the database to the minute prior to the failed migration wave.
3. **Targeted Nullification:** A provided rollback SQL script can selectively wipe imported records based on the `audit_log` or `client_uuid` signatures while preserving new data generated by early adopters.

## Testing Standards
- The migration tool must run end-to-end against a sandboxed local/staging Supabase project using a copy of real AppSheet data.
- The `06-verify.ts` script must pass 100/100 randomly sampled legacy Visit records.
- **Media Parity:** The count of successfully synchronized `visit_media` rows must equal the count of valid source photos exported from Google Drive.
